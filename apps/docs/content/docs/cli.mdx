---
title: The CLI
description: Scaffold a project, preview scenes, and render videos with @effect-motion/cli
---

The fastest way to a rendered video is the CLI — one command each for
scaffolding, previewing, and rendering, driven by two code entrypoints
(`studio.ts` and `render.ts`) instead of a config file:

```bash
pnpm create effect-motion
# or: npm create effect-motion / yarn create effect-motion / bun create effect-motion
```

The wizard asks where to create the project (`.` uses the current directory
and names the project after it), which package manager to use, and whether
to set up Biome for linting/formatting, then installs a ready-to-run
project:

```
my-project/
├─ src/
│  ├─ scenes/hello-world.ts   # a scene: a module exporting `scene`
│  ├─ assets/                 # static files
│  └─ main.ts                 # the movie — a scene composing the others
├─ studio.ts                  # what `motion studio` previews
├─ render.ts                  # what `motion render` executes
├─ AGENTS.md                  # authoring rules for AI coding agents
├─ biome.json                 # if Biome was selected
├─ package.json               # exact pins of effect-motion + effect
└─ tsconfig.json
```

Versions are pinned exactly on purpose: the `effect` pin is a determinism
invariant (upgrading effect can change seeded random sequences), so a
scaffolded project upgrades both together, deliberately.

Both entrypoints are ordinary TypeScript that **imports your scenes as
values** — which is what makes typed resources work end to end: a scene
that declares fonts or images carries that requirement in its type, and
the entrypoint won't compile until the loaders are provided. There is no
stringly-typed registry anywhere.

## Preview — studio.ts

```bash
motion studio                       # serves ./studio.ts
motion studio ./other.studio.ts     # or any entrypoint you point it at
```

`studio.ts` default-exports the studio's registration — a record of
scenes, and (once any scene declares resources) one `layers` covering all
of them:

```ts title="studio.ts"
import { studioConfig } from "@effect-motion/cli";
import { scene as main } from "./src/main";
import { scene as helloWorld } from "./src/scenes/hello-world";

export default studioConfig({
	scenes: {
		"hello-world": helloWorld, // key = unique identifier
		main: { scene: main, fps: 30 }, // per-entry player options
	},
	// layers: Layer.mergeAll(Font.layer(Pacifico, …), Image.layer(Rocket, …)),
});
```

Record keys are the picker's identifiers; the label is the scene's display
name (`Scene.make("Hello World", gen)`) when set, else the key. Only
registered scenes appear — registration is one import and one entry, and
it's what lets the compiler prove `layers` covers every scene. Hot reload
is plain Vite HMR over the entrypoint's import graph: edits, new
registrations, and removals all refresh without restarting.

Multiple studio setups are multiple files — e.g. an `experiments.studio.ts`
with different layers over the same scenes.

## Render — render.ts

```bash
motion render                 # executes ./render.ts
motion render ./teaser.render.ts
```

`render.ts` is an ordinary program: it calls the
[export API](/docs/going-further/export) and default-exports the effect.
The CLI provides the platform and runs it — nothing more:

```ts title="render.ts"
import { Video } from "@effect-motion/export";
import * as Effect from "effect/Effect";
import { scene as main } from "./src/main";

export default Effect.gen(function* () {
	yield* Video.render(main, "./output/main.mp4", {
		settings: { frameRate: 60 },
	});
	// more outputs are more calls:
	// yield* Video.render(main, "./output/main-hd.mp4", { dpr: 2 });
});
```

Rendering runs scene → ThorVG raster → bundled ffmpeg → H.264 MP4, no
system ffmpeg required; output directories are created for you. A scene
with typed resources provides its loaders right here
(`.pipe(Effect.provide(layers))`) — forgetting one is a compile error in
this file, not a runtime surprise. `--verbose` prints full error cause
chains.

There are no render flags: code is the config. Knobs you used to reach for
as flags (fps, seed, dpr, output paths) are values in `render.ts` — read
`process.env`/`process.argv` there if you want them switchable. The same
file runs without the CLI: pipe through `NodeServices` from
`@effect/platform-node` and execute it with `tsx render.ts`.

## Migrating from motion.config.ts

Each former target becomes one `Video.render` call in `render.ts`
(`<output>/<name>.mp4` is now just the path you write); the `player` blocks
become entry options in `studio.ts`; glob-discovered scenes need a one-line
registration. `defineConfig` and config discovery are gone.
