title: Randomness & frame-exactness
description: Seeded randomness, and why animations land exactly on target

Scene time is counted in **frames** rather than read from a wall clock, and
randomness comes from a **seeded** generator rather than `Math.random()`.
Together those are what let a scene be replayed, scrubbed, and exported as the
same animation you authored.

## Seeded randomness

A scene is a pure function of `(scene, settings)`. It never reads the wall clock
(durations are frame counts at the runner's frame rate) and never calls
`Math.random()`. Instead, Effect's **`Random`** service is provided to every
scene, seeded from settings:

```ts
import { Random } from "effect";

// same seed → same sequence
const x = yield* Random.nextBetween(40, 460);
const y = yield* Random.nextBetween(40, 260);
```

The default seed is the fixed string `"effect-motion"`, so a scene that scatters
shapes "randomly" scatters them the same way each run. Pass a different `seed`
in settings for a different — but equally repeatable — result.

Note the generator belongs to `effect`, so upgrading it can change the sequence
a given seed produces.

## Animations land exactly on target

Duration-based animations land their **final frame exactly** on the target value,
whatever the easing — no sub-pixel drift from accumulated rounding. Springs run
until the simulation settles, then **snap exactly** onto the target. So a value
tweened to `x: 440` *is* `440` on the last frame, and chained animations start
from a clean value.

## 3. Failures are loud

Scene-graph violations don't fail silently — they die with defects that name the
offender: a missing trait, an invalid spring, an unknown timing name, a duplicate
parent, exceeding `maxFrames`. You find out at author time, not from a subtly
wrong frame.

## Concurrency doesn't change the frames

Because concurrency runs on a per-frame barrier (see
[Scenes & the frame model](/docs/concepts/scenes)), frame *n* holds the same
values no matter how the work was scheduled — `Scene.all` and `Scene.fork`
change how a scene is written, not what it renders. `run` and `stream` agree,
and nested scenes replay like standalone runs.

Frames are reproducible; **pixels are not promised bit-identical** across GPUs,
drivers, or platforms, since rasterization and the GPU depth-of-field pass can
differ in the last bit.
