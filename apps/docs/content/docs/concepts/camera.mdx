The camera is the view onto your scene: a real 3D camera with a position, an
orientation, and a focal length. Move it and the whole scene reprojects without
any shape's data changing. It is an **ordinary instance**, so the same animators
drive it as drive everything else.

```ts
import { Motion, Scene } from "effect-motion";

const cam = yield* Scene.camera;
// dolly forward and orbit — the primitives you already know
yield* cam.pipe(
	Motion.moveTo({ z: -400 }, "2 seconds", "easeInOutCubic"),
	Motion.tweenTo({ rotY: Math.PI / 6 }, "1 second"),
);
```

- `~position` (`x`, `y`, `z`) is the camera's **world position** — `moveTo`,
  `move`, `spring` all apply. `x`/`y` pan; `z` dollies (moves toward or away).
- `rotX`, `rotY`, `rotZ` are the camera's **Euler orientation** — animate them
  with `tween` / `tweenTo` to orbit, tilt, or roll.
- `focalLength` sets the **field of view** — and with it, how strong
  perspective feels. It is measured in scene pixels, After Effects–style: the
  default is the AE 50mm-equivalent, `width × 50/36` (≈694 on a 500-wide
  scene). Smaller values are wider lenses (a given `z` moves and scales
  things more); larger values are telephoto (depth flattens out). It replaces
  the old flat `zoom` scalar.
- The camera is never drawn. A default identity camera is always present, so
  `Scene.camera` never returns null and scenes that ignore depth render
  unchanged.

World coordinates never move: a circle authored at `x: 100, z: -300` always has
that data, whatever the camera does. Projection happens at render time, so
determinism and `moveTo` semantics are untouched.

## Depth is real

Every object carries a world `z` (default 0 — the screen plane). The camera
projects `(x, y, z)` to the screen through a perspective projection, so:

- **near is bigger, far is smaller** — same world size, foreshortened by depth.
- **render order is depth-to-camera, not tree order** — a deeper object paints
  behind a nearer one no matter where it sits in the tree. Objects re-sort as
  the camera moves.
- **parallax falls out for free** — pan the camera and far objects drift slower
  than near ones, with no dedicated parallax primitive.

Cards at increasing depth scatter into the distance, a big `Rect` tilts back as
a floor, and the free camera dollies forward while orbiting — the draw order
re-sorts every frame as the camera swings around.

### Parallax from depth

Give objects (or a `Group` of them) different `z` values and pan the camera:
the far ones lag, the near ones track. This is the old "layered parallax", now
just a consequence of perspective.

<Example name="camera-parallax" />

The far starfield barely moves, the mid squares drift at half speed, and the
near circles track the camera fully — all from one camera pan and three depths.

## Tilted planes

A `Rect` carries an Euler orientation (`rotX`/`rotY`/`rotZ`). Leave it at zero
and the Rect is a camera-facing billboard; tilt it and it becomes a flat plane
in 3D — a floor, a wall, a card lying back. Its four corners are projected
individually, so a receding plane renders as a true perspective trapezoid.

```ts
// a floor: a wide Rect tilted back so it recedes toward the horizon
yield* Scene.instantiate(Shapes.Rect, {
	x: -300, y: 180, z: -200,
	width: 900, height: 900,
	rotX: Math.PI / 2.3,
	fill: "#232946",
});
```

Tilt is scoped to `Rect` for now; other shapes keep their orientation fields but
render as billboards.

## Depth of field

Two more camera fields opt into focus: `aperture` (blur strength, default `0`
— no depth of field, zero render cost) and `focusDistance` (view-space
distance to the sharp plane, defaulting to the resting distance so the `z = 0`
plane is in focus). Content blurs by how far it sits from the focus plane;
the plane itself stays pixel-sharp.

Both are plain numeric fields, so a **rack focus** — walking the sharp plane
from one subject to another — is one tween:

<Example name="rack-focus" />

Blur is applied per depth region at render time. Each blurred region costs
roughly 2 ms per frame at a 500×300 comp (≈7 ms at the default player's
high-dpi size) on the software renderer — a scene with several distinct
depths playing at 60fps will feel it; exports are unaffected. Two
approximations to know about: an entity gets one blur amount from its anchor
(a particle field spanning depth blurs uniformly), and depth of field is a
ThorVG-renderer feature like fonts and images.

## HUD: content the camera can't touch

A `Shapes.Hud` container projects its subtree through the **identity camera**
instead of the active one: titles, watermarks, and lower thirds stay bolted
to the glass while the camera dollies, shakes, or racks focus — and HUD
content is structurally exempt from depth of field (the identity camera is a
pinhole). HUD always paints on top of world content.

The container's `x`/`y` are screen coordinates and tween like anything else,
so a lower-third entrance is one tween on the container:

<Example name="hud" />

A Hud must be a top-level child of the scene — nesting it inside world
content is a loud error (world offsets composing into screen coordinates
would be incoherent). Children's `z` still works *inside* the HUD (deliberate
in-HUD parallax against the resting camera); it defaults to 0, which is
plain-2D placement. Whole sub-scenes mount into a HUD with the usual
`Scene.play(scene, { parent: hud })`.

## Swapping the camera

`Scene.camera` is the active camera. To drive the view from an instance you
create yourself (for example, to have it follow a subject), swap it in with
`Scene.setCamera`:

```ts
import { Camera } from "effect-motion";

const chase = yield* Scene.instantiate(Camera.Camera, { x: 0, y: 0, z: 1000 });
yield* Scene.setCamera(chase);
// animating `chase` now moves the view
```

Only one camera is active at a time; its live data becomes the view on every
subsequent frame. Swapping between two pre-framed cameras is an instant **cut**:

<Example name="camera-swap" />

It opens on the wide two-shot so you register both subjects first — that's the
reference frame that makes the cut readable. The default camera then pushes in on
the left subject; `Scene.setCamera` hands off to a second camera already framed
on the right subject (the instant cut), which eases back out to the wide shot.
